Test-Driven Development (TDD)

In our experience one of the more effective practices in software development is Test-Driven Development or TDD, it helps the developer to write higher quality code. Kent Beck, who is credited with having developed or 'rediscovered' the technique, stated that TDD encourages simple designs and inspires confidence. TDD is one of the practices in Extreme Programming (XP).

How it works

As the name states, the developer drives the development of your software by tests. This means that before writing any production code there needs to be a test that proves that the code will do what is expected. Or in other words, you start by writing a test and after this test you write the code which makes the test pass.

Robert C. Martin (Uncle Bob Martin) has written about the three basic rules of Test-Driven Development:

  • You are not allowed to write any production code unless it is to make a failing unit test pass.
  • You are not allowed to write any more of a unit test than is sufficient to fail; and compilation failures are failures.
  • You are not allowed to write any more production code than is sufficient to pass the one failing unit test.

Test-Driven Design or Development by Example

The emphasis on the word "test" is a bit misleading; the goal of Test-Driven Development is not to get high test coverage but lies in the fact that there has gone much more thought into writing and designing the code. Because of this people have suggested using the phrase Test-Driven Design instead of Test-Driven Development or even call the practice Development by Example.

Benefits

  • Higher overall code quality and design
  • More confidence in the code by the developers
  • Faster feedback loop to verify that no side effects where introduced
  • Faster development because of the previous points
